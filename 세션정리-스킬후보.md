## 스킬화 후보 1: Agentic MAS Phase Completion Playbook

### Trigger
- 사용자가 LangGraph/Python MAS 프로젝트를 Phase 단위로 끝까지 구현하라고 요청할 때
- backfill, evidence ledger, critic rubric, live delivery, docs, git push가 포함될 때

### Procedure
1. 현재 git/test 상태 확인
2. 기존 모듈 재사용 여부 확인
3. outputs 산출물은 Guardian scan 전까지 커밋 보류
4. Phase별 최소 검증 가능한 smoke CLI 작성
5. 각 Phase마다 테스트 1개 이상 추가
6. 전체 pytest 통과 후 docs 갱신
7. Guardian scan outputs 통과 후 `git add -f` 범위 결정
8. 최종 commit/push

### Pitfalls
- outputs/는 gitignore 대상이라 `git add -f` 필요
- force-add 전 비밀값 scan 필수
- live Gmail/Slack은 도구 권한이 아니라 repo/env 방식일 수 있음
- synthetic/live-shaped data는 실제 근거와 confidence를 명확히 분리해야 함

## 스킬화 후보 2: Whitelist Backfill Injection Workflow

### Trigger
- repo 보안상 검색 API key를 코드에 넣지 않고 외부 검색 결과 JSON을 주입해야 할 때

### Procedure
1. query_plan 생성
2. whitelist URL만 사용해 title/summary/excerpt 수준 수집
3. `outputs/search/{period}_raw_results_live.json` 저장
4. schema validation
5. `run-backfill --no-dry-run --injected-results-path ...` 실행
6. evidence/review/report/memory 산출물 검증
7. 전 기간 summary 재집계

### Verification
- evidence_count >= dynamic threshold
- source_url에 example.com 없음
- trust_score >= 4 소스 충분
- critic_score >= 8.5
- hard_fail=False

## 스킬화 후보 3: Local Chroma-Compatible VectorStore Fallback

### Trigger
- Chroma/embedding API를 바로 설치·연결하기 어렵지만 RAG 검색 인터페이스를 먼저 고정해야 할 때

### Procedure
1. VectorStore protocol 정의
2. deterministic hash embedding 구현
3. JSON persist index를 `outputs/vector_store/chroma/index.json`에 저장
4. ChromaVectorStore smoke test 작성
5. FileVectorStore fallback 유지

### Why useful
- 외부 embedding API 없이 테스트 가능
- 향후 실제 Chroma로 교체할 때 public interface 유지

## 스킬화 후보 4: Guardian-Gated Outputs Commit

### Trigger
- gitignore된 outputs/ 산출물을 force-add해서 커밋해야 할 때

### Procedure
1. `uv run python -m comp_research_mas.cli guardian-scan --path outputs/`
2. severity=pass 확인
3. block_hits=0 확인
4. 필요한 산출물만 `git add -f outputs/...`
5. 테스트 재실행
6. commit/push

### Never do
- Guardian scan 없이 `git add -f .`
- token/env/credential 파일 포함

## 삿뽀로가 알면 좋은 내용

1. 이번 Phase C의 12개월 데이터는 live-shaped whitelist title/summary 기반이다. 전문 크롤링이나 비밀 API 사용은 하지 않았다.
2. 2025-10은 B-2에서 가장 품질이 높다. Ro coverage와 critic_score가 개선되어 score 10이다.
3. Chillventa 2025 직접 근거는 확인되지 않았고, 해당 사실을 조작하지 않았다.
4. Phase D의 ChromaVectorStore는 실제 chromadb 의존성이 아니라 Chroma-compatible local JSON index다. 나중에 chromadb 설치 시 교체 가능하다.
5. Phase H의 OpenAI/Codex adapter는 실제 API 호출 전 dry-run 구조다. API key는 환경변수로만 다뤄야 한다.
6. Phase I의 PDF 파서는 라이브러리 없는 fallback 샘플이다. 실제 카탈로그 운영에는 pypdf/PyMuPDF/OCR 도입이 다음 개선점이다.
7. Slack 실제 전송은 Hermes `send_message` target `slack:업무`로 가능하다. repo 내부 webhook 방식은 `SLACK_WEBHOOK_URL`이 필요하다.
8. Gmail 실제 전송은 repo `live_sender.py` 기준 `GMAIL_APP_PASSWORD`가 필요하다.
